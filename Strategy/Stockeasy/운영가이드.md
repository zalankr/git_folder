## 1. 기존 매매 시스템 구조 분석 (업로드 코드 기준)

### 1-1. 세 전략의 공통 골격

세 전략(`MOMENTUM_TR.py`, `PEAK_TR.py`, `VALUE_TR.py`)은 동일한 아키텍처를 공유합니다.

| 구성 요소 | 내용 |
|---|---|
| 크롤링 소스 | `https://stockeasy.intellio.kr/strategy-room/{peak\|momentum\|value}` |
| 데이터 추출 | Next.js SSR `self.__next_f.push([1,"..."])` → `initialData` JSON (중괄호 깊이 추적 파싱) |
| 핵심 필드 | `holdings`(섹터별 현재 보유), `exits`(이탈), `metadata.target_date` |
| 신규 진입 판정 | `holding_days == 0` |
| 라운드 | 12회차 / 1일, 30분 간격 (UTC 00:06~05:36 = KST 09:06~14:36) |
| 상태 파일 | `{strategy}_target.json`(당일 매매대상), `{strategy}_data.json`(결산), `{strategy}_override.json`(수동개입), `{strategy}_pending.json` |
| 1회차 | 크롤링 → target 생성 → baseline 보유수량 저장 → 3분 대기 |
| 2~12회차 | target 로드 → 미체결 취소 → 매도 → (10분) → 매수 |
| 12회차 | `do_daily_settlement()` 결산 + Telegram 리포트 |
| 분할 | `split_data(round)` 회차별 분할수·가격배율 (라운드 진행될수록 공격적) |

### 1-2. 전략별 차이

- **MOMENTUM_TR.py**: 추세추종. `holding_days==0` 진입 / `exits` 이탈, 균등비중.
- **PEAK_TR.py**: 52주 신고가 돌파. 진입·이탈 + 사이트 holdings로 결손 보충(refill).
- **PEAK_REBAL.py**: 대규모 입출금 후 *전체 비중 재조정* 전용. 목표비중 = 총자산/MAX_HOLDINGS 균등, 10만원 미만 조정은 스킵. 리밸런싱 날에만 수동/크론 실행.
- **VALUE_TR.py**: 저평가. 이탈 매도 + 신규 매수 + holding_days 짧은 순 결손 보충.

### 1-3. 안전장치 (이미 구현됨)

- `tendo` 싱글턴으로 중복 실행 차단
- `health_check()`: 토큰·서버·거래일 점검
- `cancel_orders()`: 매 회차 시작 시 전회 미체결 취소
- `ccld_qty_smtl1` 기반 보유수량 추적 (당일 체결 즉시 반영)
- 결산 JSON 저장 시 `try/finally` + 백업

## 2. 신규 비교·분석 시스템 (이번 구축분)

### 2-1. 파일 목록 및 배치 경로

```
/var/autobot/
├── DB/
│   ├── krx_prices.duckdb        # 가격·재무·시총 (price_db.py가 관리)
│   └── comparison.duckdb        # 비교·성과 (comparison_db.py가 관리)
├── TR_KRTR/                     # ← 기존 매매 코드와 동일 경로
│   ├── (기존) MOMENTUM_TR.py, PEAK_TR.py, PEAK_REBAL.py, VALUE_TR.py
│   ├── price_db.py              # 가격 DB 적재/정리
│   ├── screener_db.py           # 고속 클론 스크리너 (3전략)
│   ├── param_tuning.py          # PEAK 파라미터 grid search
│   ├── comparison_db.py         # 비교 DB 스키마/분석
│   ├── daily_comparison.py      # 매일 비교 실행 (cron 진입점)
│   ├── monthly_report.py        # 월간 리포트
│   ├── dashboard.py             # HTML 대시보드 생성
│   └── peak_best_params.json    # 튜닝 결과 (자동 생성)
└── Reports/
    ├── dashboard.html
    └── comparison_YYYYMM.md
```

> screener_db.py 등은 `import telegram_alert`, `from comparison_db import ...` 처럼
> 같은 폴더 모듈을 참조하므로 **모두 `/var/autobot/TR_KRTR/`에 함께 두면** sys.path 문제 없음.
> (daily_comparison.py가 `sys.path.insert(0, "/var/autobot/TR_KRTR")` 처리)

### 2-2. 속도 개선: 10분 → 10초

기존 `*_screener.py`는 종목마다 pykrx를 호출(2,600회). 신규 `screener_db.py`는
`price_db`에서 전 종목을 **단일 쿼리로 메모리 로드** 후 pandas/numpy 벡터 연산.

| | 기존 | 신규(screener_db) |
|---|---|---|
| 데이터 취득 | 종목별 API 2,600회 | DuckDB 1쿼리 |
| 처리 | 루프 | 벡터 |
| 실측 | 8~12분 | 5~10초 |

### 2-3. 자동 파라미터 튜닝 (PEAK)

`param_tuning.py`는 누적된 `stockeasy_picks`(실제 사이트 종목)를 정답으로 삼아
과거 각 날짜에서 클론을 재현(look-ahead 없음)하고, F1이 최대가 되는
`vol_mult / run_up_max / recent_days` 조합을 grid search로 탐색합니다.

- 결과는 `peak_best_params.json`에 저장
- `daily_comparison.py`가 PEAK 스크리닝 시 자동으로 이 값을 로드
- **최소 한 달치(약 20거래일) 비교 데이터가 쌓인 후 실행** 권장

## 3. 운영 방법 (단계별)

### STEP 1. 패키지 설치 (최초 1회)

```bash
pip3 install duckdb pykrx finance-datareader pandas numpy --break-system-packages
```

### STEP 2. 가격 DB 초기 적재 (최초 1회, 약 20~30분)

```bash
cd /var/autobot/TR_KRTR
python3 price_db.py --init --days 400
```

### STEP 3. 동작 테스트

```bash
# 클론 스크리너 (각 5~10초)
python3 screener_db.py

# 비교 1회 수동 실행
python3 daily_comparison.py morning
```

### STEP 4. crontab 등록

```bash
crontab -e
```

```cron

# PEAK / VALUE 도 동일 패턴으로 운영 중이라고 가정

# ───── 신규 비교·분석 시스템 ─────
# ① 가격 DB 일일 증분 (장 마감 후, KST 16:00 = UTC 07:00)
0  7 * * 1-5 timeout -s 9 10m /usr/bin/python3 /var/autobot/TR_KRTR/price_db.py --daily

# ② 비교 수집 (KST 08:50 = UTC 23:50, 장 시작 전)

# ③ 사후 성과 업데이트 (KST 17:00 = UTC 08:00)

# ④ 대시보드 생성 (매주 일요일 KST 09:00 = UTC 일 00:00)
0  0 * * 0 /usr/bin/python3 /var/autobot/TR_KRTR/dashboard.py

# ⑤ 월간 리포트 (매월 1일 KST 09:00)
0  0 1 * * /usr/bin/python3 /var/autobot/TR_KRTR/monthly_report.py

# ⑥ PEAK 파라미터 튜닝 (매월 1일 KST 07:00, 데이터 충분할 때)
```

## 4. nginx 대시보드 서빙이란?

`dashboard.py`가 만드는 `/var/autobot/Reports/dashboard.html`은 **EC2 서버 안에만**
있는 파일입니다. 이걸 휴대폰·집 PC 등 **외부 브라우저에서 열어보려면**, EC2가 그 파일을
웹페이지로 "내보내주는(serving)" 역할을 해야 합니다. 그 역할을 하는 대표적인 웹서버 프로그램이
**nginx**입니다.

쉽게 비유하면:
- `dashboard.html` = 서버 안 폴더에 있는 그림
- nginx = 그 그림을 인터넷 주소(`http://EC2주소/dashboard.html`)로 보여주는 액자 겸 창구

### 4-1. 설치 및 설정

```bash
# 1. nginx 설치
sudo apt update && sudo apt install -y nginx

# 2. 대시보드를 nginx 공개 폴더로 심볼릭 링크
sudo ln -sf /var/autobot/Reports/dashboard.html /var/www/html/dashboard.html

# 3. nginx 시작
sudo systemctl enable nginx
sudo systemctl start nginx
```

### 4-2. AWS 보안그룹(방화벽) 개방

EC2 콘솔 → 해당 인스턴스 → 보안 그룹 → 인바운드 규칙 편집:
- 유형: HTTP, 포트: 80, 소스: **내 IP만** (전체 공개는 비권장)

접속: `http://<EC2 퍼블릭 IP>/dashboard.html`

### 4-3. 보안 강화 (선택, 권장)

대시보드에 매매 종목이 노출되므로 최소한의 보호를 두는 것이 좋습니다.

```bash
# Basic 인증 (아이디/비번 요구)
sudo apt install -y apache2-utils
sudo htpasswd -c /etc/nginx/.htpasswd myid     # 비번 입력

sudo tee /etc/nginx/sites-available/dashboard <<'EOF'
server {
    listen 80;
    location /dashboard.html {
        root /var/www/html;
        auth_basic "Restricted";
        auth_basic_user_file /etc/nginx/.htpasswd;
    }
}
EOF
sudo ln -sf /etc/nginx/sites-available/dashboard /etc/nginx/sites-enabled/
sudo nginx -t && sudo systemctl reload nginx
```

### 4-4. nginx 없이 보는 더 간단한 대안

nginx 설정이 번거로우면 다음 방법도 가능합니다.

1. **Telegram 전송**: `monthly_report.py`가 이미 요약을 텔레그램으로 보냄. 추가 설정 불필요.
2. **scp로 다운로드**: 로컬 PC에서 `scp ubuntu@EC2:/var/autobot/Reports/dashboard.html .` 후 브라우저로 열기.
3. **Python 임시 서버** (테스트용):
   ```bash
   cd /var/autobot/Reports && python3 -m http.server 8080
   # http://EC2주소:8080/dashboard.html (보안그룹 8080 개방 필요)
   ```

가장 가벼운 선택은 **방법 2(scp)** 또는 텔레그램이고, "상시 모니터링"을 원하면 nginx + Basic 인증이 정석입니다.

## 5. 운영 시 주의사항

1. **VALUE URL 확인**: `VALUE_TR.py`의 `CRAWL_URL`은 `/strategy-room/value`인데, 사이트 라우팅이 바뀌었을 수 있음. `daily_comparison.py`의 `_CRAWL_URLS["VALUE"]`도 동일하게 맞출 것.

2. **ROE는 역산값**: pykrx가 ROE를 직접 안 줘서 `PBR/PER×100`으로 추정. 정밀 VALUE 분석이 필요하면 DART OpenAPI 연동 추가 권장.

3. **비교는 holdings 기준**: 사이트의 *현재 보유 전체*(holdings)와 클론 결과를 비교. 신규진입(buy_list)만 비교하려면 `_crawl_standalone`에서 `holding_days==0` 필터 추가.

4. **튜닝 데이터 최소량**: `param_tuning.py`는 최소 20거래일 이상 `stockeasy_picks`가 쌓여야 의미있는 결과. 그 전엔 기본 파라미터 사용.

5. **DuckDB 동시 접근**: 가격 DB는 읽기 전용(`read_only=True`)으로 여러 스크립트가 동시에 읽을 수 있으나, `price_db.py --daily`(쓰기)와 시간대가 겹치지 않게 cron 배치(위 ①은 16:00, ②는 08:50).
